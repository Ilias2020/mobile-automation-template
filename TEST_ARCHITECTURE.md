# Архитектура фреймворка мобильной автоматизации

## Общая идея

Фреймворк построен для автоматизации мобильных UI-тестов Android приложений с использованием:

- Java
- Appium
- TestNG
- Gradle

Основная цель архитектуры — создать **масштабируемый и переиспользуемый шаблон**, который можно применять в разных мобильных проектах.

---

# Основные принципы архитектуры

## 1. Page Object Model (POM)

Каждый экран приложения представлен отдельным классом.

Примеры:

LoginPage — экран авторизации  
CatalogPage — экран каталога

Page Object содержит:

- локаторы элементов
- действия пользователя
- методы проверки состояния страницы

Это позволяет:

- изолировать UI-логику
- уменьшить дублирование кода
- упростить поддержку тестов

---

# Разделение ответственности

Архитектура разделена на несколько уровней.

## Test Layer

Тесты описывают только **бизнес-сценарии**.

Пример:
AuthChoicePage
→ LoginPage
→ submit
→ CatalogPage


Тест **не должен работать напрямую с локаторами**.

---

## Page Layer

Page классы описывают:

- элементы экрана
- действия пользователя
- проверки состояния страницы

Пример:
LoginPage
enterPhone()
enterPassword()
submit()


---

## BasePage

Базовый класс для всех страниц.

Содержит универсальные методы взаимодействия с UI:

- waitVisible
- waitClickable
- type
- isPresent

Это позволяет не дублировать код в каждом Page Object.

---

## Driver Layer

Управление драйвером вынесено в отдельные классы.

### DriverFactory

Создаёт Appium сессию.

Отвечает за:

- настройки устройства
- настройки Appium
- создание AndroidDriver

### DriverManager

Хранит экземпляр драйвера через ThreadLocal.

Это позволяет:

- централизованно получать driver
- поддерживать будущий параллельный запуск тестов

---

## BaseTest

Базовый класс для всех тестов.

Отвечает за жизненный цикл теста:

@BeforeMethod  
создание драйвера

@AfterMethod  
закрытие драйвера

---

# Поток выполнения теста

Тест выполняется по следующей схеме:

Test  
↓  
BaseTest  
↓  
DriverFactory создаёт драйвер  
↓  
Test вызывает Page Object  
↓  
Page Object использует BasePage методы  
↓  
Appium выполняет действия в приложении

---

# Локаторы

Для поиска элементов используются:

- accessibilityId (предпочтительно)
- resource-id
- XPath (если нет других вариантов)

Использование абсолютных XPath минимизируется.

---

# Особенности мобильной автоматизации

В процессе разработки была выявлена особенность Appium.

Метод:

hideKeyboard()

на некоторых Android устройствах может вызывать навигацию назад (Back).

Это может приводить к возврату на предыдущий экран.

Поэтому использование данного метода в тестах избегается.
