# TEST_RULES.md

# Правила написания автотестов

Данный документ описывает правила разработки автотестов в этом проекте.  
Соблюдение этих правил позволяет сохранять стабильность, читаемость и масштабируемость тестового фреймворка.

---

# Общие принципы

1. Тест должен проверять один конкретный сценарий.
2. Тест должен быть простым и читаемым.
3. Тест не должен содержать сложной логики.
4. Тест должен быть максимально независимым от других тестов.
5. Каждый тест должен запускаться отдельно.

---

# Структура теста

Каждый тест должен иметь следующую структуру:

Arrange  
подготовка данных и открытие нужной страницы

Act  
выполнение действия пользователя

Assert  
проверка результата

Пример:

AuthChoicePage authChoicePage = new AuthChoicePage();
LoginPage loginPage = authChoicePage.openLogin();

loginPage.enterPhone(phone);
loginPage.enterPassword(password);
loginPage.submit();

Assert.assertTrue(catalogPage.isOpened());

---

# Правила тестовых классов

1. Каждый тестовый класс должен наследоваться от BaseTest.
2. Тесты должны находиться в пакете tests.
3. Один класс должен тестировать один функциональный блок.

Пример:

LoginPositiveTest  
LoginNegativeTest

---

# Правила Page Object

Page Object представляет экран приложения.

Page Object должен содержать:

- локаторы элементов
- действия пользователя
- методы проверки состояния страницы

Page Object не должен содержать assert.

Проверки выполняются только в тестах.

---

# Локаторы

При поиске элементов используется следующий приоритет:

1. accessibilityId
2. resource-id
3. xpath

XPath используется только в случае отсутствия других вариантов.

Следует избегать длинных и абсолютных XPath.

---

# Ожидания элементов

Для ожиданий используются explicit wait методы.

Используются методы из BasePage:

waitVisible()  
waitClickable()

Запрещается использовать:

Thread.sleep()

---

# Работа с элементами

Все действия с элементами должны выполняться через методы BasePage:

type()  
click()  
waitVisible()  
waitClickable()

Тесты не должны напрямую использовать driver.findElement().

---

# Структура проекта

tests  
тестовые сценарии

pages  
Page Object классы

core  
управление драйвером

config  
конфигурация окружения

---

# Стабильность тестов

Тесты должны быть:

- детерминированными
- воспроизводимыми
- независимыми от предыдущих тестов

Следует избегать действий, которые могут вызвать нестабильность тестов.

Пример:

Метод hideKeyboard() может вызывать навигацию назад (Back) на некоторых Android устройствах, поэтому его использование ограничено.

---

# Развитие фреймворка

При добавлении новых тестов необходимо:

1. создавать Page Object для новых экранов
2. переиспользовать методы BasePage
3. избегать дублирования кода
